Fix multi-tab shell disambiguation by using UIA text to identify the focused tab's shell type and prefer matching process tree candidates.

Output: UIA-guided shell selection that correctly identifies the focused tab's shell in both VS Code and Windows Terminal multi-tab scenarios.

Three coordinated changes (must compile together):

**1. detect_windows.rs -- add detect_shell_type_from_uia_text()**

Add a pub function `detect_shell_type_from_uia_text(text: &str) -> Option<&'static str>` that scans UIA text lines for shell type indicators. Check the FIRST FEW lines (up to 5) since tab title appears at the top of UIA text. Patterns to match:
- Line contains "Command Prompt" (case-insensitive) -> Some("cmd")
- Line contains "Windows PowerShell" (case-insensitive) -> Some("powershell")
- Line contains "PowerShell 7" or starts with "pwsh" -> Some("pwsh")
- Line starts with "MINGW" or contains "Git Bash" -> Some("bash")
- Otherwise -> None

This function is cross-platform (pure string parsing) so tests run on all platforms. Do NOT match user@host patterns here (that's WSL territory).

**2. process.rs -- add shell_type_hint parameter to selection pipeline**

a) Add `shell_type_hint: Option<&str>` parameter to:
   - `get_foreground_info` (Windows version only, after the existing `snapshot` param)
   - `find_shell_pid` (Windows version only, after the existing `snapshot` param)
   - `find_shell_by_ancestry` (Windows version, after the existing `snapshot` param)

b) In `find_shell_by_ancestry`, after collecting candidates in each priority tier (ConPTY descendants, direct descendants, all ConPTY), before calling `pick_most_recent`:
   - If `shell_type_hint` is Some, filter candidates to those whose exe name (lowercased, .exe stripped) matches the hint
   - If filtering leaves candidates, use them; if filtering leaves ZERO, fall through to unfiltered `pick_most_recent` (don't lose the result entirely)
   - Log the filtering: `eprintln!("[process] shell_type_hint '{}': {} of {} candidates match", hint, filtered.len(), candidates.len())`

c) Non-Windows `find_shell_pid` and `get_foreground_info` signatures unchanged (no shell_type_hint param -- it's Windows-only like snapshot).

d) `find_shell_by_ancestry_no_snapshot` also gets the hint param and passes it through.

**3. mod.rs -- read UIA text first, extract hint, pass through**

In `detect_full_with_hwnd`, restructure the Windows-specific block:
- BEFORE calling `detect_app_context`, read UIA text if hwnd is available: `let uia_text = previous_hwnd.and_then(|hwnd| uia_reader::read_terminal_text_windows(hwnd).map(|t| filter::filter_sensitive(&t)));`
- Extract shell hint: `let shell_type_hint = uia_text.as_deref().and_then(detect_windows::detect_shell_type_from_uia_text);`
- Log: `eprintln!("[detect_full_with_hwnd] UIA shell_type_hint: {:?}", shell_type_hint);`
- Pass `shell_type_hint` into `detect_app_context_windows` (add parameter) which passes it to `get_foreground_info`
- The existing UIA text processing (WSL detection, visible_output assignment, infer_shell_from_text) still happens AFTER, using the already-captured `uia_text`. Do NOT read UIA text twice. Store the result and reuse it.

In `detect_app_context_windows`, add `shell_type_hint: Option<&str>` parameter and pass to `get_foreground_info`. The non-Windows `detect_app_context` signature is unchanged.

In `detect_app_context`, add the hint param for Windows cfg only. Use cfg-gated function signatures like the existing pattern for `get_foreground_info`.

IMPORTANT: The `detect_app_context` function is called from `detect_full_with_hwnd` (Windows path) and `detect_full` (non-Windows path). Use cfg-gated approach:
- On Windows: `detect_app_context(pid, pre_captured_text, shell_type_hint)`
- On non-Windows: `detect_app_context(pid, pre_captured_text)` (unchanged)
OR use a simpler approach: since `detect_full_with_hwnd` is the only Windows caller, call `detect_app_context_windows` directly instead of going through the cfg-dispatching `detect_app_context`. Check the existing code -- `detect_app_context` already cfg-dispatches to `detect_app_context_windows`, so you can either add the param to the Windows cfg path only, or bypass the dispatcher for the hwnd path.

The cleanest approach: in `detect_full_with_hwnd`'s Windows block, replace `detect_app_context(pid, pre_captured_text)` with a direct call to `detect_app_context_windows(pid, pre_captured_text, shell_type_hint)` since we're already in a `#[cfg(target_os = "windows")]` block. Add the param to `detect_app_context_windows` only. The generic `detect_app_context` and `detect_app_context_macos` stay unchanged.

Add unit tests to detect_windows.rs (or process.rs for the filtering logic) in the existing #[cfg(test)] mod.
