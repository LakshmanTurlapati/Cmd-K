<objective>
Scope UIA tree walking to terminal panel elements only, so editor content, sidebars, and menus are excluded from text capture in VS Code and Cursor.

Purpose: The current `try_walk_children` uses `TreeScope::Descendants` with `create_true_condition()`, capturing ALL text from every element in the IDE window. This causes editor content (Dockerfiles, config files with Linux paths) to leak into terminal text detection, triggering false WSL positives.
Output: New `try_scoped_terminal_walk` function that finds ControlType::List elements (xterm.js accessibility tree) and reads only their children.
</objective>

  <action>
    Add a new function `try_scoped_terminal_walk` that replaces the full-tree walk for IDE windows:

    1. Create `try_scoped_terminal_walk(automation: &UIAutomation, element: &UIElement) -> Result<String, String>`:
       a. Create a property condition for ControlType::List using `automation.create_property_condition(UIProperty::ControlType, ControlType::List.into(), None)`
       b. Use `element.find_all(TreeScope::Descendants, &list_condition)` to find all List elements
       c. For each List element found:
          - Check if it is visible using `get_property_value(UIProperty::IsOffscreen)` -- skip offscreen elements (inactive terminal tabs)
          - Get its children using `list.find_all(TreeScope::Children, &true_condition)`
          - Collect Name property from each child into a string (join with newline)
          - If the collected text looks like terminal content (has prompt patterns, command output, or is substantial -- more than just a few empty lines), return it
       d. If multiple visible List elements have terminal-like text, prefer the one with the most text (likely the active terminal)
       e. If NO List elements found or none have terminal-like text, return Err to trigger fallback

    2. Add a helper `looks_like_terminal_text(text: &str) -> bool`:
       - Returns true if text contains common terminal indicators:
         - PS prompt ("PS ", "PS>")
         - CMD prompt (single letter followed by ":\")
         - Linux prompt (contains "@" with "$" or "#")
         - Has at least 3 non-empty lines (terminal output is multi-line)
         - Contains common command output patterns (directory listings, git output, etc.)
       - Returns false for very short text or text that looks like UI labels
       - This is a best-effort heuristic, not a strict filter. The fallback covers misses.

    3. Update `read_terminal_text_inner` to use a 3-strategy approach:
       - Strategy 1: `try_text_pattern` (unchanged -- works for Windows Terminal/conhost)
       - Strategy 2: `try_scoped_terminal_walk` (NEW -- targets terminal panel in IDEs)
       - Strategy 3: `try_walk_children` (EXISTING -- full tree walk as final fallback)
       Keep `try_walk_children` as-is for the fallback. The order is: Strategy 1 -> Strategy 2 -> Strategy 3.

    4. Add eprintln! diagnostics for each strategy attempt, consistent with existing logging style:
       - "[uia_reader] trying scoped terminal walk..."
       - "[uia_reader] found N List elements, M visible"
       - "[uia_reader] scoped walk returned N chars"
       - "[uia_reader] scoped walk failed, falling back to full tree walk"

    5. Add `use uiautomation::types::{UIProperty, ControlType};` to the existing imports (under #[cfg(target_os = "windows")]).

    Important notes:
    - Do NOT remove try_walk_children -- it remains as the fallback for non-IDE terminals and edge cases
    - Do NOT modify try_text_pattern -- Strategy 1 works correctly for Windows Terminal
    - The `looks_like_terminal_text` helper does NOT need cfg(target_os = "windows") if it only does string matching -- but the callers are gated, so it should be gated too for consistency
    - Keep the existing TEXT_BUF_SIZE limit and truncate_text call in read_terminal_text_inner
