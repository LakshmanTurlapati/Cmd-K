<objective>
Fix IDE terminal shell selection so that VS Code/Cursor correctly identifies the user's active shell type instead of always picking an internal wsl.exe or cmd.exe process.

Problem: VS Code spawns 10-16+ internal wsl.exe processes (file watchers, extensions) and 5+ internal cmd.exe processes. The current `find_shell_by_ancestry` picks the highest PID among all descendant shells, which is almost always an internal wsl.exe or cmd.exe — never the user's actual PowerShell/bash/zsh terminal.

Root cause: The "highest PID = most recent = active terminal" heuristic breaks when VS Code's internal processes outnumber and out-PID user terminal processes.

Solution: For IDE terminals, apply a priority-based shell selection:
1. Prefer real interactive shells: powershell.exe, pwsh.exe, bash.exe, zsh.exe, fish.exe
2. Fall back to cmd.exe only if no interactive shell found
3. Never select wsl.exe as a "shell" — it's not a shell, and VS Code's internal wsl.exe processes are indistinguishable from terminal ones
4. Remove wsl.exe from KNOWN_TERMINAL_EXES to prevent it from being treated as a terminal

WSL detection for IDE terminals relies on existing signals:
- Window title `[WSL:]` pattern (Remote-WSL mode, from plan 01)
- UIA text Linux prompt patterns (from Phase 23)
- CWD path style fallback (from plan 01)

Output: Updated process.rs with IDE-aware shell priority, updated detect_windows.rs with wsl.exe removed from KNOWN_TERMINAL_EXES.
</objective>

In the Windows `find_shell_by_ancestry` function, after collecting `descendant_shells` and before the final "pick highest PID" logic, add IDE-aware shell prioritization.

The key insight: VS Code spawns many internal cmd.exe processes (git, tasks, extensions). The user's actual terminal is typically powershell.exe, pwsh.exe, bash.exe, zsh.exe, or fish.exe. For IDE terminals, we should prefer these "interactive shells" over cmd.exe.

**Location**: After the "descendant_shells.is_empty()" block and ConPTY fallback, BEFORE the final `descendant_shells.iter().max_by_key(...)`.

**Code to add** (insert before the final "Pick the most recently spawned" comment):

```rust
        // For IDE terminals (VS Code, Cursor), deprioritize cmd.exe.
        // VS Code spawns many internal cmd.exe processes for git, tasks, and extensions.
        // User terminal tabs are typically powershell.exe, pwsh.exe, bash.exe, etc.
        let is_ide = super::detect_windows::is_ide_with_terminal_exe(
            &super::detect_windows::get_exe_name_for_pid(app_pid_u32).unwrap_or_default()
        );
        if is_ide && descendant_shells.len() > 1 {
            let interactive: Vec<_> = descendant_shells.iter()
                .filter(|(_, name)| {
                    let lower = name.to_lowercase();
                    // Keep only real interactive shells, exclude cmd.exe (VS Code internal)
                    lower != "cmd.exe"
                })
                .collect();
            if !interactive.is_empty() {
                eprintln!("[process] IDE mode: preferring interactive shells ({} of {} descendants)",
                    interactive.len(), descendant_shells.len());
                for (pid, name) in &interactive {
                    eprintln!("[process]   interactive shell: pid={} name={}", pid, name);
                }
                return interactive.iter()
                    .max_by_key(|(pid, _)| *pid)
                    .map(|(pid, _)| *pid as i32);
            }
        }
```

**Important**: This goes AFTER the ConPTY fallback block and BEFORE the final:
```rust
        // Pick the most recently spawned (highest PID) shell
        descendant_shells.iter()
            .max_by_key(|(pid, _)| *pid)
            .map(|(pid, _)| *pid as i32)
```

The fallback (highest PID among all descendants) is preserved for:
- Non-IDE terminals (Windows Terminal, standalone powershell, etc.)
- IDE terminals with only cmd.exe descendants (no interactive shells found)
