# Phase 23.1: VS Code WSL Terminal Tab Detection via UIA - Research

**Researched:** 2026-03-09
**Domain:** Windows UI Automation, VS Code Electron accessibility tree, WSL terminal detection
**Confidence:** MEDIUM

## Summary

The core problem is that when VS Code has multiple terminals open (some WSL, some not), the app cannot distinguish which terminal tab is active. The process tree approach fails because VS Code spawns 16+ internal wsl.exe processes for file watching, Remote-WSL extensions, and other internal features -- none of which correspond to the user's active terminal tab. Adding wsl.exe to the shell list was tried and reverted (commit bcea9cd) because it caused ALL VS Code terminals to be misdetected as WSL.

The UIA text-based approach (already working for Windows Terminal) reads the visible terminal text and detects WSL from Linux prompt patterns. For Windows Terminal, UIA TextPattern returns actual terminal buffer text because Windows Terminal natively implements the UIA TextPattern provider. For VS Code (Electron/Chromium), UIA returns a different tree -- the xterm.js accessibility tree exposes terminal text through DOM elements with `role="list"` and individual row list items (`class="xterm-accessibility-tree"`), which Chromium's UIA bridge surfaces as Name properties on child elements. The existing `try_walk_children` fallback in `uia_reader.rs` already collects Name properties from descendants, so VS Code terminal text IS accessible via UIA -- but the challenge is that VS Code's UIA tree includes ALL UI chrome (menus, sidebar, status bar, editor content) alongside terminal text, not just the active terminal.

**Primary recommendation:** Enhance UIA text reading for VS Code by (1) attempting to find the terminal-specific sub-tree in VS Code's UIA element hierarchy before falling back to full-tree walking, and (2) using the VS Code window title (which shows "[WSL: Ubuntu]" when connected via Remote-WSL) as an additional WSL detection signal alongside UIA text pattern matching.

## Architecture Patterns

### Current Detection Flow (Windows)
```
hotkey handler
  -> capture HWND + PID
  -> detect_app_context_windows(pid)
     -> find_shell_by_ancestry(pid) -- walks process tree
     -> builds TerminalContext {is_wsl, shell_type, cwd, visible_output: None}
  -> detect_full_with_hwnd(pid, pre_captured, hwnd)
     -> UIA text reading via uia_reader::read_terminal_text_windows(hwnd)
     -> detect_wsl_from_text(text) -- pattern matching
     -> infer_linux_cwd_from_text(text)
     -> infer_shell_from_text(text)
```

### The VS Code Problem (Three Layers)

**Layer 1: Process tree is useless for VS Code WSL**
- VS Code spawns 16+ internal wsl.exe processes (file watching, Remote-WSL, extension host)
- Process tree walk finds cmd.exe children of Code.exe (VS Code's internal terminal host processes)
- `detect_wsl_in_ancestry(cmd.exe_pid)` walks up to Code.exe -> explorer.exe -- no wsl.exe ancestor
- Even if wsl.exe were in KNOWN_SHELL_EXES, it matches ALL VS Code instances, not just the active terminal tab

**Layer 2: UIA text is noisy for VS Code**
- Windows Terminal: UIA TextPattern returns clean terminal buffer text (native UIA provider)
- VS Code: UIA returns xterm.js accessibility tree (via Chromium's UIA bridge) mixed with ALL UI chrome
- The `try_walk_children` walker gets Name properties from menus, sidebar items, editor text, status bar, AND terminal text -- all concatenated
- WSL patterns like `user@host:~$` may be present but buried in noise

**Layer 3: No active-tab disambiguation**
- VS Code can have multiple terminal tabs (PowerShell, CMD, WSL bash, etc.)
- UIA may return text from ALL terminal instances or just the visible one
- No mechanism to determine which terminal tab is currently focused/active

### Recommended Approach: Multi-Signal Detection

```
Step 1: Window Title Check (NEW - fast, reliable for Remote-WSL)
  -> GetWindowTextW(hwnd) on VS Code window
  -> Title contains "[WSL: Ubuntu]" or "[WSL: <distro>]"
  -> If yes: strong WSL signal (Remote-WSL mode)

Step 2: UIA Focused Element (NEW - active tab detection)
  -> UIAutomation::focused_element() gets the currently focused UIA element
  -> Walk UP from focused element to find terminal-area parent
  -> Read text only from this sub-tree (avoids menu/sidebar noise)

Step 3: UIA Text Pattern Matching (EXISTING - enhanced)
  -> detect_wsl_from_text() on captured text
  -> Already works when text is clean terminal output
  -> Enhance to handle VS Code noise (Linux path patterns, prompt patterns)

Step 4: Terminal Tab Name (NEW - supplementary signal)
  -> VS Code terminal tabs have accessible names like "bash", "pwsh", "zsh"
  -> UIA tree includes tab list with individual tab elements
  -> Active tab has selection state -- Name may indicate WSL shell
```

### Pattern 1: Window Title WSL Detection
**What:** Read VS Code window title via GetWindowTextW and check for "[WSL:" pattern
**When to use:** Always for VS Code/Cursor before UIA text analysis
**Example:**
```rust
// Source: Win32 API GetWindowTextW
#[cfg(target_os = "windows")]
pub fn get_window_title(hwnd: isize) -> Option<String> {
    use windows_sys::Win32::UI::WindowsAndMessaging::GetWindowTextW;
    use windows_sys::Win32::UI::WindowsAndMessaging::GetWindowTextLengthW;

    unsafe {
        let len = GetWindowTextLengthW(hwnd as _);
        if len <= 0 { return None; }
        let mut buf = vec![0u16; (len + 1) as usize];
        let copied = GetWindowTextW(hwnd as _, buf.as_mut_ptr(), buf.len() as i32);
        if copied <= 0 { return None; }
        Some(String::from_utf16_lossy(&buf[..copied as usize]))
    }
}

fn detect_wsl_from_window_title(title: &str) -> bool {
    // VS Code Remote-WSL shows "[WSL: Ubuntu]" or "[WSL: Debian]" in title
    title.contains("[WSL:") || title.contains("WSL:")
}
```

### Anti-Patterns to Avoid
- **Adding wsl.exe to KNOWN_SHELL_EXES:** Causes ALL VS Code instances to detect as WSL (16+ wsl.exe processes for internal features). Already tried and reverted.
- **Preferring wsl.exe over cmd.exe in process tree:** Same problem -- internal wsl.exe processes are not terminal sessions.
- **Reading UIA text from full VS Code window:** Returns menus, sidebar, editor, status bar -- too noisy for reliable pattern matching.
- **Blocking on WSL subprocess calls (wsl.exe -e pwd):** Adds latency; the 750ms timeout is already tight.

## Common Pitfalls

### Pitfall 1: VS Code UIA Returns UI Chrome, Not Just Terminal
**What goes wrong:** UIA text reading walks all descendants of the VS Code window, capturing menu items ("File", "Edit", "View"), sidebar content, editor text, AND terminal text.
**Why it happens:** VS Code is an Electron app. The entire window is one HWND. Unlike Windows Terminal which has a dedicated UIA TextPattern, VS Code's UIA tree includes everything.
**How to avoid:** Use focused element or find the terminal-specific sub-tree in the UIA hierarchy. The xterm.js accessibility tree is a List control type with specific class names.
**Warning signs:** detect_wsl_from_text returns false positive/negative because of non-terminal text.

### Pitfall 2: Remote-WSL vs Local WSL Terminal
**What goes wrong:** Two different VS Code WSL scenarios have different UIA/title behaviors:
1. **Remote-WSL (VS Code server runs in WSL):** Title shows "[WSL: Ubuntu]", terminal is bash/zsh, everything is Linux
2. **Local terminal with WSL profile:** Title does NOT show "[WSL:]", terminal tab name may say "bash" or "Ubuntu (WSL)"
**Why it happens:** Remote-WSL is a full VS Code mode; local WSL terminal is just a terminal profile.
**How to avoid:** Use window title for Remote-WSL detection AND UIA text patterns for local WSL terminals. Both signals should set is_wsl=true.
**Warning signs:** Remote-WSL works but local WSL terminal tabs don't, or vice versa.

### Pitfall 3: Timing -- Overlay Steals Focus Before UIA Read
**What goes wrong:** By the time get_app_context runs, focus has moved from VS Code to the CMD+K overlay. UIA focused_element() returns the overlay, not the VS Code terminal.
**Why it happens:** Hotkey handler captures PID/HWND before toggle_overlay, but the actual UIA reading happens later in detect_full_with_hwnd.
**How to avoid:** The HWND is already captured before overlay shows. Use element_from_handle(hwnd) to root the UIA tree at the VS Code window, then find the focused-state element within that tree (not the globally focused element).
**Warning signs:** UIA text is empty or returns overlay text instead of VS Code terminal text.

### Pitfall 4: Multiple Terminal Instances in UIA Tree
**What goes wrong:** VS Code may expose multiple terminal instances in its UIA tree. Only one is visible/active, but all may have Name text.
**Why it happens:** xterm.js creates accessibility trees for each terminal instance. Chromium may keep them in the UIA tree even when not visible.
**How to avoid:** Check for the "IsOffscreen" UIA property or focus state to identify the active terminal. Alternatively, rely on the visible terminal's text being more complete/recent.
**Warning signs:** WSL detection flips between true/false as different terminal content is read.

### Pitfall 5: 750ms Timeout Pressure
**What goes wrong:** Adding more UIA tree traversal operations (focused element lookup, sub-tree walking, tab name reading) risks exceeding the 750ms timeout in detect_full_with_hwnd.
**Why it happens:** UIA COM calls can be slow, especially cross-process for Electron apps.
**How to avoid:** Window title check is fast (< 1ms). UIA focused element is fast (single call). Only fall back to tree walking if focused element approach fails. Avoid full-tree scans.
**Warning signs:** Intermittent None returns from detect_full_with_hwnd due to timeout.

### VS Code Window Title WSL Detection
```rust
/// Detect WSL from VS Code window title.
/// Remote-WSL shows "[WSL: Ubuntu]" or similar in the window title.
fn detect_wsl_from_title(title: &str) -> bool {
    // Remote-WSL: "file.rs - project [WSL: Ubuntu] - Visual Studio Code"
    title.contains("[WSL:")
}

/// Extract WSL distro name from title if present.
fn extract_wsl_distro(title: &str) -> Option<String> {
    // "[WSL: Ubuntu]" -> "Ubuntu"
    if let Some(start) = title.find("[WSL: ") {
        let after = &title[start + 6..];
        if let Some(end) = after.find(']') {
            return Some(after[..end].to_string());
        }
    }
    None
}
```

## Open Questions

1. **What exactly does VS Code's UIA tree look like for the terminal area?**
   - What we know: xterm.js creates `<div class="xterm-accessibility-tree" role="list">` with row list items. Chromium maps this to UIA List control type.
   - What's unclear: Whether UIA exposes text from ALL terminal instances or only the visible one. Whether the focused terminal's text is distinguishable.
   - Recommendation: Build and test with Accessibility Insights or Inspect.exe to map the actual UIA tree. This is a must-do before coding.

2. **Does VS Code's terminal tab list expose accessible names via UIA?**
   - What we know: VS Code terminal tabs show names like "1: bash", "1: pwsh". Screen readers can read these.
   - What's unclear: Whether the tab list and active tab are findable in the UIA tree without walking the entire hierarchy.
   - Recommendation: Investigate with Inspect.exe. If accessible, use as supplementary WSL signal.

3. **Does the UIA focused element work after overlay steals focus?**
   - What we know: HWND is captured before overlay shows. element_from_handle works with the captured HWND.
   - What's unclear: Whether the "last focused child" state is preserved in VS Code's UIA tree after the window loses focus.
   - Recommendation: Test empirically. If not preserved, rely on window title + UIA text patterns (no focused element needed).

4. **Performance budget for additional UIA calls?**
   - What we know: Current 750ms timeout. Window title read is < 1ms. UIA focused element is ~5-10ms.
   - What's unclear: How much time is left after existing UIA text reading for additional tree traversal.
   - Recommendation: Measure timing on actual VS Code window. Budget max 100ms for new operations.
