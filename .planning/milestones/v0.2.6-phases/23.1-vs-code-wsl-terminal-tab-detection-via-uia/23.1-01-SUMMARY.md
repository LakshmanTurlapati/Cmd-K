## Decisions Made
- **Removed try_focused_subtree:** VS Code does not expose xterm.js accessibility tree via UIA without screen reader mode enabled. The focused-element strategy was implemented (commit 1aeca60) but found to not work in practice, so it was removed (commit d63f909).
- **Multi-signal pivot:** Instead of relying on a single detection method, the approach layers multiple signals: window title (fastest), full UIA tree walk (existing), CWD path style (new fallback), and process tree shell children (new fallback).
- **CWD path style as WSL indicator:** If the detected CWD starts with `/home/`, `/mnt/`, or other Linux-style paths, this is treated as a WSL signal even if other detection methods fail.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] try_focused_subtree does not work without screen reader mode**
- **Found during:** Task 2 iteration
- **Issue:** VS Code's xterm.js does not expose its accessibility tree via UIA unless screen reader mode is active. The List control approach returned no useful terminal text.
- **Fix:** Removed try_focused_subtree entirely. Improved try_walk_children to collect all List elements and reject window chrome text instead.
- **Files modified:** src-tauri/src/terminal/uia_reader.rs
- **Committed in:** d63f909

**2. [Rule 2 - Missing Critical] No fallback when window title and UIA both miss WSL**
- **Found during:** Post-Task 2 testing
- **Issue:** Some IDE configurations don't show [WSL:] in title and UIA tree walk may not find WSL patterns, leaving WSL undetected.
- **Fix:** Added CWD path style fallback (Linux paths imply WSL) and process tree shell child detection (bash/zsh children of IDE process imply WSL).
- **Files modified:** src-tauri/src/terminal/mod.rs, src-tauri/src/terminal/process.rs
- **Committed in:** 6047274, db961ad
