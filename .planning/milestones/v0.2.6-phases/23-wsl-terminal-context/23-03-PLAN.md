<objective>
Fix WSL detection so it works in practice on WSL 2 systems where Linux processes are invisible to Windows process APIs.

Purpose: UAT Test 1 (blocker) showed WSL is never detected because: (1) detect_wsl_in_ancestry cannot find wsl.exe in process tree for WSL 2 Hyper-V sessions, (2) UIA text-based WSL inference is gated behind is_wsl which is already false (circular dependency), (3) infer_shell_from_text has no pattern for "user@host:path" Linux prompts. All three issues compound to make WSL detection always fail.

Output: Updated mod.rs where UIA text is the PRIMARY WSL detection mechanism, process tree ancestry is the secondary fallback.

Three changes to src-tauri/src/terminal/mod.rs, all within `#[cfg(target_os = "windows")]` gates:

**Change 1: Add `detect_wsl_from_text` function** (new function, place near `infer_linux_cwd_from_text`)

Create a function `detect_wsl_from_text(text: &str) -> bool` that checks the last 15 lines of UIA terminal text for Linux/WSL indicators. Return true if ANY of these patterns match:

- Line contains `@` AND contains `:` with a path-like segment after the colon (starts with `/` or `~`) -- this is the canonical `user@host:/path` bash prompt
- Line contains `@` AND the segment after `@` before any space/colon contains no backslash (rules out Windows paths) AND line ends with `$` or `#` -- catches prompts where UIA captures the prompt character
- Text contains paths starting with `/home/`, `/root/`, `/var/`, `/etc/`, `/usr/`, `/tmp/`, `/opt/` -- strong Linux path indicators in terminal output (not just prompts)

Important: Do NOT match on `@` alone (email addresses in terminal output would false-positive). The `@` must be combined with colon-path or Linux path evidence.

Add an `eprintln!` log: `[detect_wsl_from_text] WSL detected from UIA text: {matched_pattern}`

**Change 2: Add `user@host:` pattern to `infer_shell_from_text`**

In `infer_shell_from_text`, add a new pattern BEFORE the fish check (line 491) and AFTER the cmd check (line 487):

```rust
// Linux bash/zsh prompt: user@host:/path or user@host:~
if trimmed.contains('@') && trimmed.contains(':') {
    let colon_pos = trimmed.find(':').unwrap();
    let before_colon = &trimmed[..colon_pos];
    let after_colon = &trimmed[colon_pos + 1..];
    if before_colon.contains('@') && (after_colon.starts_with('/') || after_colon.starts_with('~') || after_colon.trim_start().starts_with('/') || after_colon.trim_start().starts_with('~')) {
        return "bash";
    }
}
```

This catches "parzival@Parzival:~", "parzival@Parzival:/home/parzival/projects", etc. The key insight is that `user@host:path` where path starts with `/` or `~` is unambiguously a Linux prompt -- Windows prompts never use this format.

**Change 3: Restructure `detect_full_with_hwnd` to break circular dependency**

In the `#[cfg(target_os = "windows")]` block of `detect_full_with_hwnd` (lines 119-153), restructure the logic AFTER UIA text is read (after `if let Some(ref text) = uia_text`):

Replace the existing `if terminal.is_wsl { ... } else if terminal.cwd.is_none() { ... }` block with:

```rust
// Step 1: Try UIA text-based WSL detection UNCONDITIONALLY
// This is the PRIMARY WSL detection mechanism.
// Process tree ancestry (detect_wsl_in_ancestry) is the fallback,
// but it fails for WSL 2 where Linux processes run in Hyper-V VM.
if !terminal.is_wsl {
    if detect_wsl_from_text(text) {
        eprintln!("[detect_full_with_hwnd] WSL detected from UIA text (process tree missed it)");
        terminal.is_wsl = true;
    }
}

// Step 2: If WSL (from either process tree or UIA text), infer Linux context
if terminal.is_wsl {
    if let Some(linux_cwd) = infer_linux_cwd_from_text(text) {
        eprintln!("[detect_full_with_hwnd] inferred Linux CWD from UIA text: {}", &linux_cwd);
        terminal.cwd = Some(linux_cwd);
    }
    let shell = infer_shell_from_text(text);
    eprintln!("[detect_full_with_hwnd] WSL shell from UIA text: {}", shell);
    terminal.shell_type = Some(shell.to_string());
} else if terminal.cwd.is_none() {
    // Non-WSL: infer shell type from visible text when process tree detection failed
    let shell = infer_shell_from_text(text);
    eprintln!("[detect_full_with_hwnd] inferred shell from UIA text: {}", shell);
    terminal.shell_type = Some(shell.to_string());
}
```

Key differences from the old code:
- `detect_wsl_from_text` runs BEFORE the `is_wsl` gate, not inside it
- When WSL is detected via text, shell is ALWAYS inferred from text (not conditional on `shell_type.is_none()`)
- The WSL CWD inference no longer requires `is_wsl` to have been set by the process tree walk
